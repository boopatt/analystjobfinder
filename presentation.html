Project Objective

The Business Problem

The Challenge

  • Job seekers spend hours manually scanning postings across multiple sites
  • Generic search returns irrelevant results — wrong seniority, location, or skill set
  • No single tool lets users query in plain English and get structured, verified role summaries

Our Solution

  • A conversational AI chatbot that understands natural language queries
  • Retrieves only from a curated, indexed database of real job postings
  • Returns structured summaries: title, description, qualifications, salary
  • Explicitly flags when no match exists — no hallucinated results

Target Domain

Single Agent · Single Tool

  • 1 agent (LangChain tool-calling) handles all user queries
  • 1 tool — the jobindex VectorSearchRetrieverTool — is the only data source
  • Feedback is written directly from the Streamlit app to a separate Delta table via the SQL Statement Execution API — the agent is not involved
  • Simple, auditable, and easy to extend

Why Not Multi-Agent or MCP?

  • Our goal is narrowly scoped: retrieve current job knowledge from one indexed source
  • A single retriever tool is sufficient — no orchestration layer needed
  • Multi-agent adds coordination overhead with no benefit for this use case
  • MCP is designed for integrating external tools and services — unnecessary here
  • Keeping it simple reduces failure points and makes evaluation easier

Agent Development · Notebook 02

Prompt Engineering Journey

❌ Original Prompt Failures

  • Hallucinated search keywords not in the user's query
  • Tool name mismatch — agent called a tool that didn't exist
  • Returned wrong seniority levels without flagging the mismatch
  • Presented irrelevant results as if they were correct matches
  • Always called the retriever — even for general knowledge questions

✅ Improved Prompt — Routing + 5 Rules

  • Routing: Classify as job-related or general knowledge before acting — only call the retriever for job queries
  • Rule 1: Always call the retriever tool for job-related queries
  • Rule 2: Use the user's exact words as the search query
  • Rule 3: Base response only on retrieved documents
  • Rule 4: Verify results match seniority, function & location
  • Rule 5: Use MISMATCH FORMAT when results don't fit

Agent Development · Notebook 02

Structured Output Formats

✅ Match Format

When retrieved results match the user's criteria:

Title: Senior Data Engineer
Company: Deloitte
Job Description: <250-word factual summary>
Qualifications: 5+ years, Python, Spark…
Salary: $120,000 – $150,000

⚠️ Mismatch Format

When no matching listings exist for the user's criteria:

I was unable to find any [user criteria] roles.

Here is what is currently available:
Title: <actual title>
Level: <actual level>
Brief Summary: <2-3 sentences>

Honest mismatch reporting prevents the agent from presenting wrong results as correct answers.

Evaluation · Notebook 03

Evaluation Design

12 Realistic Test Questions

  • Role-specific searches (visa sponsorship, remote, managerial)
  • Multi-criteria filtering (location + level + skill)
  • Salary and experience-level questions
  • Out-of-scope requests (images, company culture)

MLflow Evaluation Framework

  • mlflow.genai.evaluate() with MLflow 3.0
  • LLM-as-judge scoring via Guidelines scorer API
  • All results logged to the JobFinder_Eval experiment
  • Per-question trace logged for full transparency

Four Scorers

  • job_relevant — response grounded in real postings?
  • filter_aware — user-specified filters acknowledged?
  • RelevanceToQuery — built-in MLflow scorer
  • Safety — built-in MLflow scorer

Why Custom Scorers?

  • Generic scorers can't evaluate domain-specific failures
  • job_relevant catches hallucination specific to job search
  • filter_aware catches the silent-filter-ignore failure we saw in testing

Evaluation · Key Decision

Why We Did Not Fine-Tune

❌ Fine-Tuning Would Not Help

  • Fine-tuning changes how a model responds — not what it knows
  • Our problem was a data gap: missing job listings, not wrong behavior
  • Job postings change frequently — static weights become stale immediately
  • 30 PDFs is far too small for meaningful fine-tuning
  • Llama 4 Maverick would require significant GPU resources to retrain

The Right Fixes

  • Expand the dataset — add entry-level & junior job listings
  • Add metadata fields — seniority, location, required skills
  • Hybrid search — combine semantic similarity with structured filters
  • Increase max_tokens — from 500 to 1000+ to prevent truncation

Evaluation told us exactly what was wrong and why — that is evaluation working correctly.

Reflections

Key Lessons Learned

Technical

  • RAG grounding is more important than model size for factual accuracy
  • Prompt engineering has a higher ROI than fine-tuning for structured tasks
  • Evaluation must be domain-specific — generic scorers miss real failure modes
  • Honest mismatch reporting builds more user trust than confident wrong answers

Process

  • Start with a small, well-scoped dataset before scaling up
  • Test with realistic queries from the actual use case, not toy examples
  • Evaluation results should drive scope decisions, not just confirm success
  • Feedback collection from users is the best source of future training data
